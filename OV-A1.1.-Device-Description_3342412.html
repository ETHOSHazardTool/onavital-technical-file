1. General Description

ONAVITAL by Onalabs Inno-hub SL is a medical-grade device for remote, spot check monitoring of vital signs. ONAVITAL provides accurate and precise measurements of Pulse Rate (PR), Oxygent Saturation (SpO2), Systolic and Diastolic Arterial Blood Pressure (ABP) and Skin-surface Temperature (ST) through a wearable physical bracelet.

ONAVITAL is intended to be used by patients that are not considered to be in a critical condition.

ONAVITAL is a wearable physical sensor bracelet worn by the patient/user including sensors and calibration algorithms for vital signs acquisition and adapters for data digitalization.

Physical wearable sensor bracelet

The wearable is comprised of a hardware electronic element, encapsulated in a bio-compatible plastic injection casing and a hypoallergenic wrist band.

The firmware works in real time using an integrated solid memory, multiplexed design of I2C signal input ports, and a serial digital communication system for sensors integrated in the device. The real time capability of the physical sensors and firmware in parallel enables the capture of clinical parameters at the same time as the biometric signals from which they are extracted are acquired.

In addition to the measurement and signal processing systems, the wearable includes elements for its correct operation: a rechargeable battery and operation commands (for example start, stop, connect). There is in memory data storage protocol to avoid its loss in case of disconnection from the communication network and embedded algorithms to be able to process the data obtained from the different sensors. The sensors give a digitised electrical response correlated with different biometric and/or physiological elements. These signals are processed to get more compact data to facilitate their transmission in near real time for sufficient functionality and operability due to the clinical characteristics of its application.

The device collects data which is then securely transmitted through an associated mobile application to the cloud-based platform.

3. Accessories

The following accessories are provided:

  • Charging cable. The magnetic charge cable is used for ONAVITAL battery charging

  • Reset pin. The reset pin is used to activate the inner button of the device for different functions explained on the user manual.

  • IOS and Android apps for visualisation. The application enables users to visualise the data captured by the wearable component.

  • Cloud based platform used to collect and throughput data from the wearable bracelet to the mobile application for visualisation. Data from the wearable is securely transmitted via an associated mobile phone to a cloud-based platform for storage and visualisation. Data securely held on the platform is then displayed by the mobile application to an individual user, or can be collated into summary dashboards or shared via APIs. The platform functions as a medical device data system and is used to manage authentication and associate device data with registered individuals.

  • IOS and Android apps for visualisation. The application enables users to visualise the data captured by the wearable component.

Mobile Application (IOs and Android)

The mobile application which runs on IOS and Android operating systems enables 1. data visualisation and 2. data transfer using bluetooth low-energy (BLE). Data visualisation occurs through the application, and the app includes adapters that allow data from the wearable bracelet to be securely transferred to a cloud-based platform and external services using HTTP REST or MQTT.

Cloud-based platform:

The cloud based platform is used to manage a registry of wearable bracelets, associate bracelets with individuals, provide authentication to individuals accessing the ONAVITAL application, and act as a data repository. Data from wearable bracelets is associated with individuals on the cloud-based platform and the platform has functionality to visualise data and make available data reports through APIs or cloud-based dashboard.

The ONAVITAL device does not include animal or human tissue or blood products or derivatives thereof or products that would be medicinal substances or use prohibited substances as defined in the Essential Requirements nor constitute a machine or personal protection equipment. There are no sterility requirements for the device.

4. Detailed Description

Glossary:

Raw Data: Data collected by the sensors directly related to physiological elements of the user / patient.
Data Feature: Parameters extracted from the Raw Data in specific periods of time and with specific periodicity. These specific items are adjusted depending on the Health Feature to be monitored.
Health Feature: Related to Vital Signs, such as Pulse Rate, Oxygen Saturation. This information is extracted from one or several Data Features.

The wearable hardware consists of integrated circuits (ICs) architected on a 4 layers of FR4 Printed Circuit Board (PCB), which are controlled by a firmware. The device architecture is designed to support:

  1. Versatility: It allows adapting to the very specific needs of each patient without increasing the operating cost.

  2. Precision: Allows the devices to know when precise signals are obtained from the sensors.

  3. Autonomy: Tasks are distributed among several sub-tasks, reducing the computational and operational needs of each of them. Using specific sub-tasks for specific monitoring need reduces battery consumption.

The wearable hardware consists of four modules:
• Control module: operational control and data management.
• Sensors and computation module: raw data extraction from the sensors, data feature computation and algorithms to measure health features.
• Communications module: management of bluetooth communications, services and security handling.
• Power module: management and control of the battery.

The hardware electronic element has embedded firmware.

A firmware operation protocol establishes the base and associated characteristics of the firmware integrated in the device divided into three parts for design purposes:

a) data processing algorithm,

b) device operating control system,

c) communications protocol.

Raw data is extracted from measurable physiological phenomena through the sensors located in the device. Health Features parameters are calculated through a series of data features extracted from the raw data.

Data is transmitted from the wearable to a companion mobile application on a device that has been connected with the bracelet and then securely transmitted to the cloud platform. The platform re-serves data to the mobile application for data visualisation when an individual accesses the relevant application functionality.

The firmware provides metadata information for user and data traceability. The metadata consists of the following parameters: Device ID (DID), Device Status (DSTATUS), Battery Status (BSTATUS), Date (DATE) and Time (TIME). Data from wearable bracelets is associated with individuals on the cloud-based platform and the platform based on records of bracelets assigned to usernames.

Mobile Application (IOS and Android)

The wearable bracelet is accompanied by a mobile application which can be used on either iOS or Android phones. The mobile application allows individuals to:

  • enter username and password.

When a user logs in, an access token and a refresh token are provided by the server. The refresh token is used to generate a new token for the API call when the access token has expired. This allows the user to keep the session open while the data is being collected, either with the application open or running in background.

  • review the last health feature measurements.

Mobile applications also pair with the wearable bracelet and data is transmitted from the wearable to the platform by enabling device pairing. The application holds data in a local cache if the platform is not accessible. Data stored in the local cache on the phone are not visualised through the application, except the last value received for each health feature. All health feature measurements visible through the mobile applications are resourced from the cloud platform.

Platform

The cloud-based platform is used to maintain a register of wearable bracelets, authorised individuals. assign bracelets to accounts and received from bracelets is associated with individual usernames to create personally identifiable data on the platform. The platform has no medical device functionality and acts as a medical device data system. It manages authentication using JSON Web Token technology and local encrypted storage to facilitate data exchange and visualisation.

  • User Authentication: When a user logs in to the app, their credentials are sent to the server for verification. Once the server verifies the user's credentials, it generates a JWT token.

  • Token Generation: The server creates a JWT by constructing the header, payload, and signature. The payload often includes the user's ID and their access rights.

  • Token Response: The server sends the JWT back to the client (the app) in the response. The client then stores this token securely.

  • Token Usage: For subsequent requests, the app includes the JWT in the Authorization header of the HTTP request, allowing the server to authenticate and authorise the user based on the information contained in the token.

  • Token Verification: Upon receiving a request, the server extracts the JWT from the Authorization header and verifies its signature using the server's secret key. It also checks the token's validity, expiration, and any other required criteria.

  • Response: If the token is valid and passes all checks, the server processes the request and responds accordingly. If the token is invalid or has expired, the server denies access and returns an appropriate error message.

The refresh token is used to generate a new token to call the API call when the access token has expired. This allows the user to keep the session open while the data is being collected, either with the application open or running in background. Tokens are stored in the operating system key chain when running of iOS and encrypted in the local application folders for Android phones. The encryption key is stored in the Android key chain.
