1. Purpose

To document product data flows. Data is managed and stored in a Cloud service provided by BioT, who is a certified ISO13485 and IEC 62304. BioT clients enjoy Business Associate Agreement (BAA) agreements with BioT that offload their cyber security and privacy liability.

2.1 Personal Data

  • E-mail login

  • Password

  • Acceptance of informed consent

  • User ID (UID)

2.2 Physiological Parameters

  • Pulse Rate

  • Oxygen Saturation

  • Systolic blood pressure

  • Skin temperature

  • Time in hours:minutes:seconds

3. Data Flows.

4. Data Privacy.

BioT is an AWS Advanced Technology Partner. As part of the mandatory requirements to be eligible for the Advanced status, BioT was required to qualify for the AWS Well-Architected Review, which is the deepest audit AWS conducts on its partners. The audit focuses on security best practices. BioT qualified for the review already at the first attempt with almost unprecedented recorded grades and compliments.

4.1 Data Access Control

Access to data resources is managed per user and per device, based on attribute-based access control. This security mechanism is used to prevent patient from creating other patients or accessing Protected Health Information (PHI) that they are not allowed to access. PHI information access is incorporated into the permission system and it is possible to define per user whether they are entitled to access PHI info and what info can they access.

The attributes to set the pertinent permissions are set up by linking the unique User ID (UID) with the unique Device ID (DID) and the credentials used to create the User. The UID is created automatically during User Sign Up and all the manufactured devices are already registered with a DID on the platform. During Sign Up on the APP, the User will be notified to introduce a DID Identification Code found inside the device packaging, linking in that manner the different attributes for access control.

4.1.1 Device Authentication

Devices are authenticated using X.509 certificates. The system performs both device authentication with the server, as well as server authentication with the device. Device keys are pre-loaded during the manufacturing process and are tied to a unique Device ID. In some cases devices can’t be loaded with a key during the manufacturing process. In these cases, the association is done ad-hoc in the field and requires a privileged logged in user to ensure reliance.

4.1.2 Secure Connectivity

All communication with the devices is encrypted over TLS. All communication to the IoT broker is done using MQTT, all other protocols are blocked and all ports are closed.

4.1.3 User Authentication

User authentication is done by means of JSON Web Token (JWT) technology. The tokens are obtained when logging in, and this information is saved in the application’s Secure Storage (flutter library that saves data encrypted). The response of the API call at the time of login returns the information in JSON format consisting of 3 parts:

  • General ID information.

  • Access JWT information.

  • Refresh JWT information.

4.2 Users Audit

Every change made by a user is logged into an audit log. Changes to PHI data are logged and can be traced back directly to the user that instigated the change.

4.3 Security Services

All system data is managed per region and adheres to regional regulations. It is possible to define where data is stored (what region) and where it can be sent to, providing the owner of the system full control over sensitive issues like keeping medical data within the country they are working in.

BioT Multi Region automates routing of data from the device to the cloud in a way that complies with the privacy regulation set at the country of the owner of the device.

5. Data Resiliency

The system includes a high availability architecture based on AWS high availability zones that aims to provide continuation of service as well as data availability. The system high availability zones ensure that high availability instances are always running on different locations so a malfunction/problem in one location of AWS will not create an availability problem.
