2. Source Code and Version Control

The code of the application is uploaded on GitHub for version control. GitHub is a collaborative development platform which is used to store and manage the application’s source code, as well as control the changes made over time.

Main functionalities:

  • Code Repository: GitHub provides a centralized repository where the application's source code is stored. The repository acts as an online depot where the entire history of code changes and versions is kept.

  • Version Control: GitHub utilizes the Git version control system, which allows for precise tracking of every change made to the code over time. Each modification, addition, or deletion of files is recorded as a "commit."

  • Branches: Git and GitHub enable the creation of separate branches from the main code to work on new features or fixes without affecting the stable or main version of the application. This facilitates collaboration and parallel development of different functionalities.

  • Pull Requests: When working in a team, developers can create pull requests to discuss and review the changes made in a branch before incorporating them into the main code. This allows for reviewing and commenting on proposed changes, collaborating with other team members, and ensuring code quality.

  • Issues and Issue Tracking: GitHub also provides functionality for managing issues, bugs, or feature requests. Team members can report problems, assign tasks, make comments, and track the progress of solutions.

  • Collaboration and Access Control: GitHub facilitates collaboration among developers by allowing them to clone the repository, work on their local copies, and then push the changes back to the central repository. Additionally, access permissions can be defined to control who can contribute to the repository and who has read-only access.

By using GitHub for version control, several benefits are obtained:

  • Complete History of Changes: Any previous version of the code can be easily accessed and reviewed, facilitating the identification and correction of errors.

  • Quality Control: Pull requests and comments enable continuous code review and improvement, helping to maintain high quality in the application's development.

In summary, by uploading the code of an application to GitHub, precise control of changes, effective collaboration, and a platform for sharing knowledge and contributing to software development are obtained more efficiently.

4. Architecture

The ONAVITAL application is based on the hexagonal architecture, and it’s organised by folders, where:

  • APP: In this folder is where the UI is programmed.

  • CONTEXTS: In this folder is where the functionalities are programmed, and it is applied the Vertical Slicing technic.

  • TEST_DRIVER: In this folder is where the integration tests are programmed.

Mobile applications are also used to transmit data from the wearable to the platform by enabling device pairing and holding data in a local cache if the platform is not accessible. Data stored in the local cache on the phone are not visualised through the application. All health feature measurements visible through the mobile applications are resourced from the BioT platform.

The ONAVITAL mobile applications are based on an hexagonal architecture, also known as ports and adapters or the onion architecture. This is a software architectural pattern that promotes a modular and flexible design for applications. It aims to de-couple the core business logic from external dependencies, such as frameworks, databases, or external services, allowing the application to be more maintainable, testable, and adaptable to changes.

The mobile applications are constructed with the following concentric layers, each with a specific responsibility.

  • Core/Application Layer: This contains the domain model, business rules, and use cases.The primary focus is the implementation of the business rules and domain-specific operations.

  • Ports/Interface Layer: This layer defines the interfaces or ports that allow communication between the application core and the external world. Ports are contracts that specify the services required by the core. Typically, these are defined as interfaces or abstract classes. The core does not depend on concrete implementations of these ports; they are implemented by external adapters.

  • Adapters Layer: Adapters are responsible for implementing the ports defined in the interface layer and act as intermediaries between the application and the external dependencies, such as frameworks, databases, web services, or UI. Adapters convert the data received from external systems into a format that the application core understands and vice versa.

  • Inbound Adapters: are responsible for handling requests and input from external systems and translating them into a format suitable for the application core.

  • Outbound Adapters: provide the necessary abstractions to interact with external systems, such as databases, third-party services, or messaging systems. They encapsulate the implementation details and provide a simple interface for the core to communicate with these external systems.

Encrypted Local Storage

An encryption library from Flutter, such as flutter_secure_storage, is used to encrypt and store sensitive data locally. These libraries provide secure storage solutions by encrypting data using algorithms like AES (Advanced Encryption Standard) and securely storing them in the device's file system.

Bluetooth Low Energy Security

Bluetooth Low Energy (BLE) features a series of stages to ensure data security and integrity.

  • Generation of identifier code: The ID code is generated by the remote device, not the controller, independently of any other BLE devices present (1). This allows communication to only occur when this code is accessed via device hardware access.

  • Connectivity and pairing between devices: It has some differences in security aspects compared to some security features such as Secure Simple Pairing (2). The pairing models are similar to Secure Simple Pairing from the user's perspective and have the same names, but differ in the quality of protection provided. The partnership models present for BLE are Just Works, Out of Band and Passkey Entry (3).

  • Data encryption: The data transmitted via BLE is encrypted using AES-CCM cryptography (4).

  • Data authentication: Bluetooth LE supports the ability to send authenticated data via ATT encryption between two devices with a trusted / associated relationship. This is done by authenticating the data with a Connection Signature Resolution Key (CSRK) (5).

  • Privacy: BLE supports a feature to reduce the ability to track a BLE device over a period of time, using an autonomous dynamic system that changes the address of the BLE device frequently and in an automated manner (6).
